11     # Form helpers are designed to make working with resources much easier
 
12     # compared to using vanilla HTML.
 
13     #
 
14     # Forms for models are created with +form_for+. That method yields a form
 
15     # builder that knows the model the form is about. The form builder is thus
 
16     # able to generate default values for input fields that correspond to model
 
17     # attributes, and also convenient names, IDs, endpoints, etc.
 
18     #
 
19     # Conventions in the generated field names allow controllers to receive form
 
20     # data nicely structured in +params+ with no effort on your side.
 
21     #
 
22     # For example, to create a new person you typically set up a new instance of
 
23     # +Person+ in the <tt>PeopleController#new</tt> action, <tt>@person</tt>, and
 
24     # pass it to +form_for+:
 
25     #
 
26     #   <%= form_for @person do |f| %>
 
27     #     <%= f.label :first_name %>:
 
28     #     <%= f.text_field :first_name %><br />
 
29     #
 
30     #     <%= f.label :last_name %>:
 
31     #     <%= f.text_field :last_name %><br />
 
32     #
 
33     #     <%= f.submit %>
 
34     #   <% end %>
 
35     #
 
36     # The HTML generated for this would be (modulus formatting):
 
37     #
 
38     #   <form action="/people" class="new_person" id="new_person" method="post">
 
39     #     <div style="margin:0;padding:0;display:inline">
 
40     #       <input name="authenticity_token" type="hidden" value="NrOp5bsjoLRuK8IW5+dQEYjKGUJDe7TQoZVvq95Wteg=" />
 
41     #     </div>
 
42     #     <label for="person_first_name">First name</label>:
 
43     #     <input id="person_first_name" name="person[first_name]" size="30" type="text" /><br />
 
44     #
 
45     #     <label for="person_last_name">Last name</label>:
 
46     #     <input id="person_last_name" name="person[last_name]" size="30" type="text" /><br />
 
47     #
 
48     #     <input id="person_submit" name="commit" type="submit" value="Create Person" />
 
49     #   </form>
 
50     #
 
51     # As you see, the HTML reflects knowledge about the resource in several spots,
 
52     # like the path the form should be submitted to, or the names of the input fields.
 
53     #
 
54     # In particular, thanks to the conventions followed in the generated field names, the
 
55     # controller gets a nested hash <tt>params[:person]</tt> with the person attributes
 
56     # set in the form. That hash is ready to be passed to <tt>Person.create</tt>:
 
57     #
 
58     #   if @person = Person.create(params[:person])
 
59     #     # success
 
60     #   else
 
61     #     # error handling
 
62     #   end
 
63     #
 
64     # Interestingly, the exact same view code in the previous example can be used to edit
 
65     # a person. If <tt>@person</tt> is an existing record with name "John Smith" and ID 256,
 
66     # the code above as is would yield instead:
 
67     #
 
68     #   <form action="/people/256" class="edit_person" id="edit_person_256" method="post">
 
69     #     <div style="margin:0;padding:0;display:inline">
 
70     #       <input name="_method" type="hidden" value="put" />
 
71     #       <input name="authenticity_token" type="hidden" value="NrOp5bsjoLRuK8IW5+dQEYjKGUJDe7TQoZVvq95Wteg=" />
 
72     #     </div>
 
73     #     <label for="person_first_name">First name</label>:
 
74     #     <input id="person_first_name" name="person[first_name]" size="30" type="text" value="John" /><br />
 
75     #
 
76     #     <label for="person_last_name">Last name</label>:
 
77     #     <input id="person_last_name" name="person[last_name]" size="30" type="text" value="Smith" /><br />
 
78     #
 
79     #     <input id="person_submit" name="commit" type="submit" value="Update Person" />
 
80     #   </form>
 
81     #
 
82     # Note that the endpoint, default values, and submit button label are tailored for <tt>@person</tt>.
 
83     # That works that way because the involved helpers know whether the resource is a new record or not,
 
84     # and generate HTML accordingly.
 
85     #
 
86     # The controller would receive the form data again in <tt>params[:person]</tt>, ready to be
 
87     # passed to <tt>Person#update_attributes</tt>:
 
88     #
 
89     #   if @person.update_attributes(params[:person])
 
90     #     # success
 
91     #   else
 
92     #     # error handling
 
93     #   end
 
94     #
 
95     # That's how you tipically work with resources.

102       # Creates a form and a scope around a specific model object that is used
 
103       # as a base for questioning about values for the fields.
 
104       #
 
105       # Rails provides succinct resource-oriented form generation with +form_for+
 
106       # like this:
 
107       #
 
108       #   <%= form_for @offer do |f| %>
 
109       #     <%= f.label :version, 'Version' %>:
 
110       #     <%= f.text_field :version %><br />
 
111       #     <%= f.label :author, 'Author' %>:
 
112       #     <%= f.text_field :author %><br />
 
113       #   <% end %>
 
114       #
 
115       # There, +form_for+ is able to generate the rest of RESTful form
 
116       # parameters based on introspection on the record, but to understand what
 
117       # it does we need to dig first into the alternative generic usage it is
 
118       # based upon.
 
119       #
 
120       # === Generic form_for
 
121       #
 
122       # The generic way to call +form_for+ yields a form builder around a
 
123       # model:
 
124       #
 
125       #   <%= form_for :person do |f| %>
 
126       #     <%= f.error_messages %>
 
127       #     First name: <%= f.text_field :first_name %><br />
 
128       #     Last name : <%= f.text_field :last_name %><br />
 
129       #     Biography : <%= f.text_area :biography %><br />
 
130       #     Admin?    : <%= f.check_box :admin %><br />
 
131       #   <% end %>
 
132       #
 
133       # There, the argument is a symbol or string with the name of the
 
134       # object the form is about.
 
135       #
 
136       # The form builder acts as a regular form helper that somehow carries the
 
137       # model. Thus, the idea is that
 
138       #
 
139       #   <%= f.text_field :first_name %>
 
140       #
 
141       # gets expanded to
 
142       #
 
143       #   <%= text_field :person, :first_name %>
 
144       #
 
145       # The rightmost argument to +form_for+ is an
 
146       # optional hash of options:
 
147       #
 
148       # * <tt>:url</tt> - The URL the form is submitted to. It takes the same
 
149       #   fields you pass to +url_for+ or +link_to+. In particular you may pass
 
150       #   here a named route directly as well. Defaults to the current action.
 
151       # * <tt>:html</tt> - Optional HTML attributes for the form tag.
 
152       #
 
153       # Worth noting is that the +form_for+ tag is called in a ERb evaluation
 
154       # block, not an ERb output block. So that's <tt><% %></tt>, not
 
155       # <tt><%= %></tt>.
 
156       #
 
157       # Also note that +form_for+ doesn't create an exclusive scope. It's still
 
158       # possible to use both the stand-alone FormHelper methods and methods
 
159       # from FormTagHelper. For example:
 
160       #
 
161       #   <%= form_for @person do |f| %>
 
162       #     First name: <%= f.text_field :first_name %>
 
163       #     Last name : <%= f.text_field :last_name %>
 
164       #     Biography : <%= text_area :person, :biography %>
 
165       #     Admin?    : <%= check_box_tag "person[admin]", @person.company.admin? %>
 
166       #   <% end %>
 
167       #
 
168       # This also works for the methods in FormOptionHelper and DateHelper that
 
169       # are designed to work with an object as base, like
 
170       # FormOptionHelper#collection_select and DateHelper#datetime_select.
 
171       #
 
172       # === Resource-oriented style
 
173       #
 
174       # As we said above, in addition to manually configuring the +form_for+
 
175       # call, you can rely on automated resource identification, which will use
 
176       # the conventions and named routes of that approach. This is the
 
177       # preferred way to use +form_for+ nowadays.
 
178       #
 
179       # For example, if <tt>@post</tt> is an existing record you want to edit
 
180       #
 
181       #   <%= form_for @post do |f| %>
 
182       #     ...
 
183       #   <% end %>
 
184       #
 
185       # is equivalent to something like:
 
186       #
 
187       #   <%= form_for @post, :as => :post, :url => post_path(@post), :html => { :method => :put, :class => "edit_post", :id => "edit_post_45" } do |f| %>
 
188       #     ...
 
189       #   <% end %>
 
190       #
 
191       # And for new records
 
192       #
 
193       #   <%= form_for(Post.new) do |f| %>
 
194       #     ...
 
195       #   <% end %>
 
196       #
 
197       # is equivalent to something like:
 
198       #
 
199       #   <%= form_for @post, :as => :post, :url => post_path(@post), :html => { :class => "new_post", :id => "new_post" } do |f| %>
 
200       #     ...
 
201       #   <% end %>
 
202       #
 
203       # You can also overwrite the individual conventions, like this:
 
204       #
 
205       #   <%= form_for(@post, :url => super_post_path(@post)) do |f| %>
 
206       #     ...
 
207       #   <% end %>
 
208       #
 
209       # If you have an object that needs to be represented as a different
 
210       # parameter, like a Client that acts as a Person:
 
211       #
 
212       #   <%= form_for(@post, :as => :client do |f| %>
 
213       #     ...
 
214       #   <% end %>
 
215       #
 
216       # And for namespaced routes, like +admin_post_url+:
 
217       #
 
218       #   <%= form_for([:admin, @post]) do |f| %>
 
219       #    ...
 
220       #   <% end %>
 
221       #
 
222       # === Unobtrusive JavaScript
 
223       # 
224       # Specifying:  
225       #  
226       #    :remote => true
 
227       #
 
228       # in the options hash creates a form that will allow the unobtrusive JavaScript drivers to modify its
 
229       # behaviour. The expected default behaviour is an XMLHttpRequest in the background instead of the regular 
230       # POST arrangement, but ultimately the behaviour is the choice of the JavaScript driver implementor.
 
231       # Even though it's using JavaScript to serialize the form elements, the form submission will work just like 
232       # a regular submission as viewed by the receiving side (all elements available in <tt>params</tt>).
 
233       #
 
234       # Example:
 
235       #
 
236       #   <%= form_for(@post, :remote => true) do |f| %>
 
237       #     ...
 
238       #   <% end %>
 
239       #
 
240       # The HTML generated for this would be:
 
241       #
 
242       #   <form action='http://www.example.com' method='post' data-remote='true'>
 
243       #     <div style='margin:0;padding:0;display:inline'>
 
244       #       <input name='_method' type='hidden' value='put' />
 
245       #     </div>
 
246       #     ...
 
247       #   </form>
 
248       #
 
249       # === Customized form builders
 
250       #
 
251       # You can also build forms using a customized FormBuilder class. Subclass
 
252       # FormBuilder and override or define some more helpers, then use your
 
253       # custom builder. For example, let's say you made a helper to
 
254       # automatically add labels to form inputs.
 
255       #
 
256       #   <%= form_for @person, :url => { :action => "create" }, :builder => LabellingFormBuilder do |f| %>
 
257       #     <%= f.text_field :first_name %>
 
258       #     <%= f.text_field :last_name %>
 
259       #     <%= text_area :person, :biography %>
 
260       #     <%= check_box_tag "person[admin]", @person.company.admin? %>
 
261       #   <% end %>
 
262       #
 
263       # In this case, if you use this:
 
264       #
 
265       #   <%= render :partial => f %>
 
266       #
 
267       # The rendered template is <tt>people/_labelling_form</tt> and the local
 
268       # variable referencing the form builder is called
 
269       # <tt>labelling_form</tt>.
 
270       #
 
271       # The custom FormBuilder class is automatically merged with the options
 
272       # of a nested fields_for call, unless it's explicitely set.
 
273       #
 
274       # In many cases you will want to wrap the above in another helper, so you
 
275       # could do something like the following:
 
276       #
 
277       #   def labelled_form_for(record_or_name_or_array, *args, &proc)
 
278       #     options = args.extract_options!
 
279       #     form_for(record_or_name_or_array, *(args << options.merge(:builder => LabellingFormBuilder)), &proc)
 
280       #   end
 
281       #
 
282       # If you don't need to attach a form to a model instance, then check out
 
283       # FormTagHelper#form_tag.
 
284       def form_for(record_or_name_or_array, *args, &proc)
 
285         raise ArgumentError, "Missing block" unless block_given?
 
286 
287         options = args.extract_options!
 
288 
289         case record_or_name_or_array
 
290         when String, Symbol
 
291           ActiveSupport::Deprecation.warn("Using form_for(:name, @resource) is deprecated. Please use form_for(@resource, :as => :name) instead.", caller) unless args.empty?

332       # Creates a scope around a specific model object like form_for, but
 
333       # doesn't create the form tags themselves. This makes fields_for suitable
 
334       # for specifying additional model objects in the same form.
 
335       #
 
336       # === Generic Examples
 
337       #
 
338       #   <%= form_for @person do |person_form| %>
 
339       #     First name: <%= person_form.text_field :first_name %>
 
340       #     Last name : <%= person_form.text_field :last_name %>
 
341       #
 
342       #     <%= fields_for @person.permission do |permission_fields| %>
 
343       #       Admin?  : <%= permission_fields.check_box :admin %>
 
344       #     <% end %>
 
345       #   <% end %>
 
346       #
 
347       # ...or if you have an object that needs to be represented as a different
 
348       # parameter, like a Client that acts as a Person:
 
349       #
 
350       #   <%= fields_for :person, @client do |permission_fields| %>
 
351       #     Admin?: <%= permission_fields.check_box :admin %>
 
352       #   <% end %>
 
353       #
 
354       # ...or if you don't have an object, just a name of the parameter:
 
355       #
 
356       #   <%= fields_for :person do |permission_fields| %>
 
357       #     Admin?: <%= permission_fields.check_box :admin %>
 
358       #   <% end %>
 
359       #
 
360       # Note: This also works for the methods in FormOptionHelper and
 
361       # DateHelper that are designed to work with an object as base, like
 
362       # FormOptionHelper#collection_select and DateHelper#datetime_select.
 
363       #
 
364       # === Nested Attributes Examples
 
365       #
 
366       # When the object belonging to the current scope has a nested attribute
 
367       # writer for a certain attribute, fields_for will yield a new scope
 
368       # for that attribute. This allows you to create forms that set or change
 
369       # the attributes of a parent object and its associations in one go.
 
370       #
 
371       # Nested attribute writers are normal setter methods named after an
 
372       # association. The most common way of defining these writers is either
 
373       # with +accepts_nested_attributes_for+ in a model definition or by
 
374       # defining a method with the proper name. For example: the attribute
 
375       # writer for the association <tt>:address</tt> is called
 
376       # <tt>address_attributes=</tt>.
 
377       #
 
378       # Whether a one-to-one or one-to-many style form builder will be yielded
 
379       # depends on whether the normal reader method returns a _single_ object
 
380       # or an _array_ of objects.
 
381       #
 
382       # ==== One-to-one
 
383       #
 
384       # Consider a Person class which returns a _single_ Address from the
 
385       # <tt>address</tt> reader method and responds to the
 
386       # <tt>address_attributes=</tt> writer method:
 
387       #
 
388       #   class Person
 
389       #     def address
 
390       #       @address
 
391       #     end
 
392       #
 
393       #     def address_attributes=(attributes)
 
394       #       # Process the attributes hash
 
395       #     end
 
396       #   end
 
397       #
 
398       # This model can now be used with a nested fields_for, like so:
 
399       #
 
400       #   <%= form_for @person do |person_form| %>
 
401       #     ...
 
402       #     <%= person_form.fields_for :address do |address_fields| %>
 
403       #       Street  : <%= address_fields.text_field :street %>
 
404       #       Zip code: <%= address_fields.text_field :zip_code %>
 
405       #     <% end %>
 
406       #   <% end %>
 
407       #
 
408       # When address is already an association on a Person you can use
 
409       # +accepts_nested_attributes_for+ to define the writer method for you:
 
410       #
 
411       #   class Person < ActiveRecord::Base
 
412       #     has_one :address
 
413       #     accepts_nested_attributes_for :address
 
414       #   end
 
415       #
 
416       # If you want to destroy the associated model through the form, you have
 
417       # to enable it first using the <tt>:allow_destroy</tt> option for
 
418       # +accepts_nested_attributes_for+:
 
419       #
 
420       #   class Person < ActiveRecord::Base
 
421       #     has_one :address
 
422       #     accepts_nested_attributes_for :address, :allow_destroy => true
 
423       #   end
 
424       #
 
425       # Now, when you use a form element with the <tt>_destroy</tt> parameter,
 
426       # with a value that evaluates to +true+, you will destroy the associated
 
427       # model (eg. 1, '1', true, or 'true'):
 
428       #
 
429       #   <%= form_for @person do |person_form| %>
 
430       #     ...
 
431       #     <%= person_form.fields_for :address do |address_fields| %>
 
432       #       ...
 
433       #       Delete: <%= address_fields.check_box :_destroy %>
 
434       #     <% end %>
 
435       #   <% end %>
 
436       #
 
437       # ==== One-to-many
 
438       #
 
439       # Consider a Person class which returns an _array_ of Project instances
 
440       # from the <tt>projects</tt> reader method and responds to the
 
441       # <tt>projects_attributes=</tt> writer method:
 
442       #
 
443       #   class Person
 
444       #     def projects
 
445       #       [@project1, @project2]
 
446       #     end
 
447       #
 
448       #     def projects_attributes=(attributes)
 
449       #       # Process the attributes hash
 
450       #     end
 
451       #   end
 
452       #
 
453       # This model can now be used with a nested fields_for. The block given to
 
454       # the nested fields_for call will be repeated for each instance in the
 
455       # collection:
 
456       #
 
457       #   <%= form_for @person do |person_form| %>
 
458       #     ...
 
459       #     <%= person_form.fields_for :projects do |project_fields| %>
 
460       #       <% if project_fields.object.active? %>
 
461       #         Name: <%= project_fields.text_field :name %>
 
462       #       <% end %>
 
463       #     <% end %>
 
464       #   <% end %>
 
465       #
 
466       # It's also possible to specify the instance to be used:
 
467       #
 
468       #   <%= form_for @person do |person_form| %>
 
469       #     ...
 
470       #     <% @person.projects.each do |project| %>
 
471       #       <% if project.active? %>
 
472       #         <%= person_form.fields_for :projects, project do |project_fields| %>
 
473       #           Name: <%= project_fields.text_field :name %>
 
474       #         <% end %>
 
475       #       <% end %>
 
476       #     <% end %>
 
477       #   <% end %>
 
478       #
 
479       # Or a collection to be used:
 
480       #
 
481       #   <%= form_for @person do |person_form| %>
 
482       #     ...
 
483       #     <%= person_form.fields_for :projects, @active_projects do |project_fields| %>
 
484       #       Name: <%= project_fields.text_field :name %>
 
485       #     <% end %>
 
486       #   <% end %>
 
487       #
 
488       # When projects is already an association on Person you can use
 
489       # +accepts_nested_attributes_for+ to define the writer method for you:
 
490       #
 
491       #   class Person < ActiveRecord::Base
 
492       #     has_many :projects
 
493       #     accepts_nested_attributes_for :projects
 
494       #   end
 
495       #
 
496       # If you want to destroy any of the associated models through the
 
497       # form, you have to enable it first using the <tt>:allow_destroy</tt>
 
498       # option for +accepts_nested_attributes_for+:
 
499       #
 
500       #   class Person < ActiveRecord::Base
 
501       #     has_many :projects
 
502       #     accepts_nested_attributes_for :projects, :allow_destroy => true
 
503       #   end
 
504       #
 
505       # This will allow you to specify which models to destroy in the
 
506       # attributes hash by adding a form element for the <tt>_destroy</tt>
 
507       # parameter with a value that evaluates to +true+
 
508       # (eg. 1, '1', true, or 'true'):
 
509       #
 
510       #   <%= form_for @person do |person_form| %>
 
511       #     ...
 
512       #     <%= person_form.fields_for :projects do |project_fields| %>
 
513       #       Delete: <%= project_fields.check_box :_destroy %>
 
514       #     <% end %>
 
515       #   <% end %>

700       # Returns a checkbox tag tailored for accessing a specified attribute (identified by +method+) on an object
 
701       # assigned to the template (identified by +object+). This object must be an instance object (@object) and not a local object.
 
702       # It's intended that +method+ returns an integer and if that integer is above zero, then the checkbox is checked.
 
703       # Additional options on the input tag can be passed as a hash with +options+. The +checked_value+ defaults to 1
 
704       # while the default +unchecked_value+ is set to 0 which is convenient for boolean values.
 
705       #
 
706       # ==== Gotcha
 
707       #
 
708       # The HTML specification says unchecked check boxes are not successful, and
 
709       # thus web browsers do not send them. Unfortunately this introduces a gotcha:
 
710       # if an +Invoice+ model has a +paid+ flag, and in the form that edits a paid
 
711       # invoice the user unchecks its check box, no +paid+ parameter is sent. So,
 
712       # any mass-assignment idiom like
 
713       #
 
714       #   @invoice.update_attributes(params[:invoice])
 
715       #
 
716       # wouldn't update the flag.
 
717       #
 
718       # To prevent this the helper generates an auxiliary hidden field before
 
719       # the very check box. The hidden field has the same name and its
 
720       # attributes mimick an unchecked check box.
 
721       #
 
722       # This way, the client either sends only the hidden field (representing
 
723       # the check box is unchecked), or both fields. Since the HTML specification
 
724       # says key/value pairs have to be sent in the same order they appear in the
 
725       # form, and parameters extraction gets the last occurrence of any repeated
 
726       # key in the query string, that works for ordinary forms.
 
727       #
 
728       # Unfortunately that workaround does not work when the check box goes
 
729       # within an array-like parameter, as in
 
730       #
 
731       #   <%= fields_for "project[invoice_attributes][]", invoice, :index => nil do |form| %>
 
732       #     <%= form.check_box :paid %>
 
733       #     ...
 
734       #   <% end %>
 
735       #
 
736       # because parameter name repetition is precisely what Rails seeks to distinguish
 
737       # the elements of the array. For each item with a checked check box you
 
738       # get an extra ghost item with only that attribute, assigned to "0".
 
739       #
 
740       # In that case it is preferable to either use +check_box_tag+ or to use
 
741       # hashes instead of arrays.
 
742       #
 
743       # ==== Examples
 
744       #   # Let's say that @post.validated? is 1:
 
745       #   check_box("post", "validated")
 
746       #   # => <input name="post[validated]" type="hidden" value="0" />
 
747       #   #    <input type="checkbox" id="post_validated" name="post[validated]" value="1" />
 
748       #
 
749       #   # Let's say that @puppy.gooddog is "no":
 
750       #   check_box("puppy", "gooddog", {}, "yes", "no")
 
751       #   # => <input name="puppy[gooddog]" type="hidden" value="no" />
 
752       #   #    <input type="checkbox" id="puppy_gooddog" name="puppy[gooddog]" value="yes" />
 
753       #
 
754       #   check_box("eula", "accepted", { :class => 'eula_check' }, "yes", "no")
 
755       #   # => <input name="eula[accepted]" type="hidden" value="no" />
 
756       #   #    <input type="checkbox" class="eula_check" id="eula_accepted" name="eula[accepted]" value="yes" />
 
757       #
 
758       def check_box(object_name, method, options = {}, checked_value = "1", unchecked_value = "0")
 
759         InstanceTag.new(object_name, method, self, options.delete(:object)).to_check_box_tag(options, checked_value, unchecked_value)
 
760       end

1177       # Add the submit button for the given form. When no value is given, it checks
 
1178       # if the object is a new resource or not to create the proper label:
 
1179       #
 
1180       #   <%= form_for @post do |f| %>
 
1181       #     <%= f.submit %>
 
1182       #   <% end %>
 
1183       # 
1184       # In the example above, if @post is a new record, it will use "Create Post" as
 
1185       # submit button label, otherwise, it uses "Update Post".
 
1186       #
 
1187       # Those labels can be customized using I18n, under the helpers.submit key and accept
 
1188       # the %{model} as translation interpolation:
 
1189       #
 
1190       #   en:
 
1191       #     helpers:
 
1192       #       submit:
 
1193       #         create: "Create a %{model}"
 
1194       #         update: "Confirm changes to %{model}"

1197       #

1199       #     helpers:

1201       #         post:
