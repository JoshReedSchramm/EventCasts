54   # == Inheritable callback queues
 
55   #
 
56   # Besides the overwritable callback methods, it's also possible to register callbacks through the use of the callback macros.
 
57   # Their main advantage is that the macros add behavior into a callback queue that is kept intact down through an inheritance
 
58   # hierarchy. Example:
 
59   #
 
60   #   class Topic < ActiveRecord::Base
 
61   #     before_destroy :destroy_author
 
62   #   end
 
63   #
 
64   #   class Reply < Topic
 
65   #     before_destroy :destroy_readers
 
66   #   end
 
67   #
 
68   # Now, when <tt>Topic#destroy</tt> is run only +destroy_author+ is called. When <tt>Reply#destroy</tt> is run, both +destroy_author+ and
 
69   # +destroy_readers+ are called. Contrast this to the situation where we've implemented the save behavior through overwriteable
 
70   # methods:
 
71   #
 
72   #   class Topic < ActiveRecord::Base
 
73   #     def before_destroy() destroy_author end
 
74   #   end
 
75   #
 
76   #   class Reply < Topic
 
77   #     def before_destroy() destroy_readers end
 
78   #   end
 
79   #
 
80   # In that case, <tt>Reply#destroy</tt> would only run +destroy_readers+ and _not_ +destroy_author+. So, use the callback macros when
 
81   # you want to ensure that a certain callback is called for the entire hierarchy, and use the regular overwriteable methods
 
82   # when you want to leave it up to each descendant to decide whether they want to call +super+ and trigger the inherited callbacks.
 
83   #
 
84   # *IMPORTANT:* In order for inheritance to work for the callback queues, you must specify the callbacks before specifying the
 
85   # associations. Otherwise, you might trigger the loading of a child before the parent has registered the callbacks and they won't
 
86   # be inherited.
 
87   #

93   # eval methods are deprecated.

185   # == The +after_find+ and +after_initialize+ exceptions
 
186   #
 
187   # Because +after_find+ and +after_initialize+ are called for each object found and instantiated by a finder, such as <tt>Base.find(:all)</tt>, we've had
 
188   # to implement a simple performance constraint (50% more speed on a simple test case). Unlike all the other callbacks, +after_find+ and
 
189   # +after_initialize+ will only be run if an explicit implementation is defined (<tt>def after_find</tt>). In that case, all of the
 
190   # callback types will be called.

192   # == <tt>before_validation*</tt> returning statements
 
193   #
 
194   # If the returning value of a +before_validation+ callback can be evaluated to +false+, the process will be aborted and <tt>Base#save</tt> will return +false+.
 
195   # If Base#save! is called it will raise a ActiveRecord::RecordInvalid exception.
 
196   # Nothing will be appended to the errors object.
 
197   #
 
198   # == Canceling callbacks
 
199   #
 
200   # If a <tt>before_*</tt> callback returns +false+, all the later callbacks and the associated action are cancelled. If an <tt>after_*</tt> callback returns
 
201   # +false+, all the later callbacks are cancelled. Callbacks are generally run in the order they are defined, with the exception of callbacks
 
202   # defined as methods on the model, which are called last.
 
203   #
 
204   # == Transactions
 
205   #
 
206   # The entire callback chain of a +save+, <tt>save!</tt>, or +destroy+ call runs
 
207   # within a transaction. That includes <tt>after_*</tt> hooks. If everything
 
208   # goes fine a COMMIT is executed once the chain has been completed.
 
209   #
 
210   # If a <tt>before_*</tt> callback cancels the action a ROLLBACK is issued. You
 
211   # can also trigger a ROLLBACK raising an exception in any of the callbacks,
 
212   # including <tt>after_*</tt> hooks. Note, however, that in that case the client
 
213   # needs to be aware of it because an ordinary +save+ will raise such exception
 
214   # instead of quietly returning +false+.
