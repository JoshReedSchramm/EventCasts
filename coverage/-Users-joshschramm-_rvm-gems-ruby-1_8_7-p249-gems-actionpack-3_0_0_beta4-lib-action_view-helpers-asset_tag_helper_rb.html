19     # === Using asset hosts
 
20     #
 
21     # By default, Rails links to these assets on the current host in the public
 
22     # folder, but you can direct Rails to link to assets from a dedicated asset
 
23     # server by setting ActionController::Base.asset_host in the application
 
24     # configuration, typically in <tt>config/environments/production.rb</tt>.
 
25     # For example, you'd define <tt>assets.example.com</tt> to be your asset
 
26     # host this way:
 
27     #
 
28     #   ActionController::Base.asset_host = "assets.example.com"
 
29     #
 
30     # Helpers take that into account:
 
31     #
 
32     #   image_tag("rails.png")
 
33     #   # => <img alt="Rails" src="http://assets.example.com/images/rails.png?1230601161" />
 
34     #   stylesheet_link_tag("application")
 
35     #   # => <link href="http://assets.example.com/stylesheets/application.css?1232285206" media="screen" rel="stylesheet" type="text/css" />
 
36     #
 
37     # Browsers typically open at most two simultaneous connections to a single
 
38     # host, which means your assets often have to wait for other assets to finish
 
39     # downloading. You can alleviate this by using a <tt>%d</tt> wildcard in the
 
40     # +asset_host+. For example, "assets%d.example.com". If that wildcard is
 
41     # present Rails distributes asset requests among the corresponding four hosts
 
42     # "assets0.example.com", ..., "assets3.example.com". With this trick browsers
 
43     # will open eight simultaneous connections rather than two.
 
44     #
 
45     #   image_tag("rails.png")
 
46     #   # => <img alt="Rails" src="http://assets0.example.com/images/rails.png?1230601161" />
 
47     #   stylesheet_link_tag("application")
 
48     #   # => <link href="http://assets2.example.com/stylesheets/application.css?1232285206" media="screen" rel="stylesheet" type="text/css" />
 
49     #
 
50     # To do this, you can either setup four actual hosts, or you can use wildcard
 
51     # DNS to CNAME the wildcard to a single asset host. You can read more about
 
52     # setting up your DNS CNAME records from your ISP.
 
53     #
 
54     # Note: This is purely a browser performance optimization and is not meant
 
55     # for server load balancing. See http://www.die.net/musings/page_load_time/
 
56     # for background.
 
57     #
 
58     # Alternatively, you can exert more control over the asset host by setting
 
59     # +asset_host+ to a proc like this:
 
60     #
 
61     #   ActionController::Base.asset_host = Proc.new { |source|
 
62     #     "http://assets#{source.hash % 2 + 1}.example.com"
 
63     #   }
 
64     #   image_tag("rails.png")
 
65     #   # => <img alt="Rails" src="http://assets1.example.com/images/rails.png?1230601161" />

74     # absolute path of the asset with any extensions and timestamps in place,
 
75     # for example "/images/rails.png?1230601161".
 
76     #
 
77     #    ActionController::Base.asset_host = Proc.new { |source|

111     # === Customizing the asset path
 
112     #
 
113     # By default, Rails appends asset's timestamps to all asset paths. This allows
 
114     # you to set a cache-expiration date for the asset far into the future, but
 
115     # still be able to instantly invalidate it by simply updating the file (and
 
116     # hence updating the timestamp, which then updates the URL as the timestamp
 
117     # is part of that, which in turn busts the cache).
 
118     #
 
119     # It's the responsibility of the web server you use to set the far-future
 
120     # expiration date on cache assets that you need to take advantage of this
 
121     # feature. Here's an example for Apache:
 
122     #
 
123     #   # Asset Expiration
 
124     #   ExpiresActive On
 
125     #   <FilesMatch "\.(ico|gif|jpe?g|png|js|css)$">
 
126     #     ExpiresDefault "access plus 1 year"
 
127     #   </FilesMatch>
 
128     #
 
129     # Also note that in order for this to work, all your application servers must
 
130     # return the same timestamps. This means that they must have their clocks
 
131     # synchronized. If one of them drifts out of sync, you'll see different
 
132     # timestamps at random and the cache won't work. In that case the browser
 
133     # will request the same assets over and over again even thought they didn't
 
134     # change. You can use something like Live HTTP Headers for Firefox to verify
 
135     # that the cache is indeed working.
 
136     #
 
137     # This strategy works well enough for most server setups and requires the
 
138     # least configuration, but if you deploy several application servers at
 
139     # different times - say to handle a temporary spike in load - then the
 
140     # asset time stamps will be out of sync. In a setup like this you may want
 
141     # to set the way that asset paths are generated yourself.
 
142     #
 
143     # Altering the asset paths that Rails generates can be done in two ways.
 
144     # The easiest is to define the RAILS_ASSET_ID environment variable. The
 
145     # contents of this variable will always be used in preference to
 
146     # calculated timestamps. A more complex but flexible way is to set
 
147     # <tt>ActionController::Base.config.asset_path</tt> to a proc
 
148     # that takes the unmodified asset path and returns the path needed for
 
149     # your asset caching to work. Typically you'd do something like this in
 
150     # <tt>config/environments/production.rb</tt>:
 
151     #
 
152     #   # Normally you'd calculate RELEASE_NUMBER at startup.
 
153     #   RELEASE_NUMBER = 12345
 
154     #   config.action_controller.asset_path_template = proc { |asset_path|
 
155     #     "/release-#{RELEASE_NUMBER}#{asset_path}"
 
156     #   }
 
157     #
 
158     # This example would cause the following behaviour on all servers no
 
159     # matter when they were deployed:
 
160     #
 
161     #   image_tag("rails.png")
 
162     #   # => <img alt="Rails" src="/release-12345/images/rails.png" />
 
163     #   stylesheet_link_tag("application")
 
164     #   # => <link href="/release-12345/stylesheets/application.css?1232285206" media="screen" rel="stylesheet" type="text/css" />
 
165     #
 
166     # Changing the asset_path does require that your web servers have
 
167     # knowledge of the asset template paths that you rewrite to so it's not
 
168     # suitable for out-of-the-box use. To use the example given above you
 
169     # could use something like this in your Apache VirtualHost configuration:
 
170     #
 
171     #   <LocationMatch "^/release-\d+/(images|javascripts|stylesheets)/.*$">
 
172     #     # Some browsers still send conditional-GET requests if there's a
 
173     #     # Last-Modified header or an ETag header even if they haven't
 
174     #     # reached the expiry date sent in the Expires header.
 
175     #     Header unset Last-Modified
 
176     #     Header unset ETag
 
177     #     FileETag None
 
178     #
 
179     #     # Assets requested using a cache-busting filename should be served
 
180     #     # only once and then cached for a really long time. The HTTP/1.1
 
181     #     # spec frowns on hugely-long expiration times though and suggests
 
182     #     # that assets which never expire be served with an expiration date
 
183     #     # 1 year from access.
 
184     #     ExpiresActive On
 
185     #     ExpiresDefault "access plus 1 year"
 
186     #   </LocationMatch>
 
187     #
 
188     #   # We use cached-busting location names with the far-future expires
 
189     #   # headers to ensure that if a file does change it can force a new
 
190     #   # request. The actual asset filenames are still the same though so we
 
191     #   # need to rewrite the location from the cache-busting location to the
 
192     #   # real asset location so that we can serve it.
 
193     #   RewriteEngine On
 
194     #   RewriteRule ^/release-\d+/(images|javascripts|stylesheets)/(.*)$ /$1/$2 [L]

332       def javascript_include_tag(*sources)
 
333         options = sources.extract_options!.stringify_keys
 
334         concat  = options.delete("concat")
 
335         cache   = concat || options.delete("cache")
 
336         recursive = options.delete("recursive")
 
337 
338         if concat || (config.perform_caching && cache)
 
339           joined_javascript_name = (cache == true ? "all" : cache) + ".js"
 
340           joined_javascript_path = File.join(joined_javascript_name[/^#{File::SEPARATOR}/] ? config.assets_dir : config.javascripts_dir, joined_javascript_name)
 
341 
342           unless config.perform_caching && File.exists?(joined_javascript_path)
 
343             write_asset_file_contents(joined_javascript_path, compute_javascript_paths(sources, recursive))
 
344           end
 
345           javascript_src_tag(joined_javascript_name, options)
 
346         else
 
347           sources = expand_javascript_sources(sources, recursive)
 
348           ensure_javascript_sources!(sources) if cache
 
349           sources.collect { |source| javascript_src_tag(source, options) }.join("\n").html_safe
