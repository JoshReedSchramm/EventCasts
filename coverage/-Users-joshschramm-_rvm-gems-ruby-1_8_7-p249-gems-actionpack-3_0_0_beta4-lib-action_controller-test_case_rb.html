194   # == Basic example
 
195   #
 
196   # Functional tests are written as follows:
 
197   # 1. First, one uses the +get+, +post+, +put+, +delete+ or +head+ method to simulate
 
198   #    an HTTP request.
 
199   # 2. Then, one asserts whether the current state is as expected. "State" can be anything:
 
200   #    the controller's HTTP response, the database contents, etc.
 
201   #
 
202   # For example:
 
203   #
 
204   #   class BooksControllerTest < ActionController::TestCase
 
205   #     def test_create
 
206   #       # Simulate a POST response with the given HTTP parameters.
 
207   #       post(:create, :book => { :title => "Love Hina" })

210   #       # the created book's URI.
 
211   #       assert_response :found

214   #       assert_not_nil Book.find_by_title("Love Hina")
 
215   #     end
 
216   #   end

219   #
 
220   # ActionController::TestCase will also automatically provide the following instance
 
221   # variables for use in the tests:
 
222   #
 
223   # <b>@controller</b>::
 
224   #      The controller instance that will be tested.
 
225   # <b>@request</b>::
 
226   #      An ActionController::TestRequest, representing the current HTTP
 
227   #      request. You can modify this object before sending the HTTP request. For example,
 
228   #      you might want to set some session properties before sending a GET request.
 
229   # <b>@response</b>::

232   #      after calling +post+. If the various assert methods are not sufficient, then you

248   # == Testing controller internals
 
249   #
 
250   # In addition to these specific assertions, you also have easy access to various collections that the regular test/unit assertions
 
251   # can be used against. These collections are:
 
252   #
 
253   # * assigns: Instance variables assigned in the action that are available for the view.
 
254   # * session: Objects being saved in the session.
 
255   # * flash: The flash objects currently in the session.
 
256   # * cookies: Cookies being sent to the user on this request.
 
257   #
 
258   # These collections can be used just like any other hash:
 
259   #
 
260   #   assert_not_nil assigns(:person) # makes sure that a @person instance variable was set
 
261   #   assert_equal "Dave", cookies[:name] # makes sure that a cookie called :name was set as "Dave"
 
262   #   assert flash.empty? # makes sure that there's nothing in the flash
 
263   #
 
264   # For historic reasons, the assigns hash uses string-based keys. So assigns[:person] won't work, but assigns["person"] will. To
 
265   # appease our yearning for symbols, though, an alternative accessor has been devised using a method call instead of index referencing.
 
266   # So assigns(:person) will work just like assigns["person"], but again, assigns[:person] will not work.
 
267   #
 
268   # On top of the collections, you have the complete url that a given action redirected to available in redirect_to_url.
 
269   #
 
270   # For redirects within the same controller, you can even call follow_redirect and the redirect will be followed, triggering another
 
271   # action call which can then be asserted against.
 
272   #
