4   # Layouts reverse the common pattern of including shared headers and footers in many templates to isolate changes in
 
5   # repeated setups. The inclusion pattern has pages that look like this:
 
6   #
 
7   #   <%= render "shared/header" %>
 
8   #   Hello World
 
9   #   <%= render "shared/footer" %>
 
10   #
 
11   # This approach is a decent way of keeping common structures isolated from the changing content, but it's verbose
 
12   # and if you ever want to change the structure of these two includes, you'll have to change all the templates.
 
13   #
 
14   # With layouts, you can flip it around and have the common structure know where to insert changing content. This means
 
15   # that the header and footer are only mentioned in one place, like this:
 
16   #
 
17   #   // The header part of this layout
 
18   #   <%= yield %>
 
19   #   // The footer part of this layout
 
20   #
 
21   # And then you have content pages that look like this:
 
22   #
 
23   #    hello world
 
24   #
 
25   # At rendering time, the content page is computed and then inserted in the layout, like this:
 
26   #
 
27   #   // The header part of this layout
 
28   #   hello world
 
29   #   // The footer part of this layout
 
30   #
 
31   # == Accessing shared variables
 
32   #
 
33   # Layouts have access to variables specified in the content pages and vice versa. This allows you to have layouts with
 
34   # references that won't materialize before rendering time:
 
35   #
 
36   #   <h1><%= @page_title %></h1>
 
37   #   <%= yield %>
 
38   #
 
39   # ...and content pages that fulfill these references _at_ rendering time:
 
40   #
 
41   #    <% @page_title = "Welcome" %>
 
42   #    Off-world colonies offers you a chance to start a new life
 
43   #
 
44   # The result after rendering is:
 
45   #
 
46   #   <h1>Welcome</h1>
 
47   #   Off-world colonies offers you a chance to start a new life
 
48   #
 
49   # == Layout assignment
 
50   #
 
51   # You can either specify a layout declaratively (using the #layout class method) or give
 
52   # it the same name as your controller, and place it in <tt>app/views/layouts</tt>.
 
53   # If a subclass does not have a layout specified, it inherits its layout using normal Ruby inheritance.
 
54   #
 
55   # For instance, if you have PostsController and a template named <tt>app/views/layouts/posts.html.erb</tt>,
 
56   # that template will be used for all actions in PostsController and controllers inheriting
 
57   # from PostsController.
 
58   #
 
59   # If you use a module, for instance Weblog::PostsController, you will need a template named
 
60   # <tt>app/views/layouts/weblog/posts.html.erb</tt>.
 
61   #
 
62   # Since all your controllers inherit from ApplicationController, they will use
 
63   # <tt>app/views/layouts/application.html.erb</tt> if no other layout is specified
 
64   # or provided.
 
65   #
 
66   # == Inheritance Examples
 
67   #
 
68   #   class BankController < ActionController::Base
 
69   #     layout "bank_standard"
 
70   #
 
71   #   class InformationController < BankController

90   # == Types of layouts
 
91   #
 
92   # Layouts are basically just regular templates, but the name of this template needs not be specified statically. Sometimes
 
93   # you want to alternate layouts depending on runtime information, such as whether someone is logged in or not. This can
 
94   # be done either by specifying a method reference as a symbol or using an inline method (as a proc).
 
95   #
 
96   # The method reference is the preferred approach to variable layouts and is used like this:
 
97   #
 
98   #   class WeblogController < ActionController::Base
 
99   #     layout :writers_and_readers
 
100   #
 
101   #     def index
 
102   #       # fetching posts
 
103   #     end
 
104   #
 
105   #     private
 
106   #       def writers_and_readers
 
107   #         logged_in? ? "writer_layout" : "reader_layout"
 
108   #       end
 
109   #
 
110   # Now when a new request for the index action is processed, the layout will vary depending on whether the person accessing
 
111   # is logged in or not.
 
112   #
 
113   # If you want to use an inline method, such as a proc, do something like this:
 
114   #
 
115   #   class WeblogController < ActionController::Base
 
116   #     layout proc{ |controller| controller.logged_in? ? "writer_layout" : "reader_layout" }
 
117   #
 
118   # Of course, the most common way of specifying a layout is still just as a plain template name:
 
119   #
 
120   #   class WeblogController < ActionController::Base
 
121   #     layout "weblog_standard"
 
122   #
 
123   # If no directory is specified for the template name, the template will by default be looked for in <tt>app/views/layouts/</tt>.
 
124   # Otherwise, it will be looked up relative to the template root.
 
125   #
 
126   # == Conditional layouts
 
127   #
 
128   # If you have a layout that by default is applied to all the actions of a controller, you still have the option of rendering
 
129   # a given action or set of actions without a layout, or restricting a layout to only a single action or a set of actions. The
 
130   # <tt>:only</tt> and <tt>:except</tt> options can be passed to the layout call. For example:
 
131   #
 
132   #   class WeblogController < ActionController::Base
 
133   #     layout "weblog_standard", :except => :rss
 
134   #
 
135   #     # ...
 
136   #
 
137   #   end
 
138   #
 
139   # This will assign "weblog_standard" as the WeblogController's layout  except for the +rss+ action, which will not wrap a layout
 
140   # around the rendered view.
 
141   #
 
142   # Both the <tt>:only</tt> and <tt>:except</tt> condition can accept an arbitrary number of method references, so
 
143   # #<tt>:except => [ :rss, :text_only ]</tt> is valid, as is <tt>:except => :rss</tt>.
 
144   #
 
145   # == Using a different layout in the action render call
 
146   #
 
147   # If most of your actions use the same layout, it makes perfect sense to define a controller-wide layout as described above.
 
148   # Sometimes you'll have exceptions where one action wants to use a different layout than the rest of the controller.
 
149   # You can do this by passing a <tt>:layout</tt> option to the <tt>render</tt> call. For example:
 
150   #
 
151   #   class WeblogController < ActionController::Base
 
152   #     layout "weblog_standard"
 
153   #
 
154   #     def help
 
155   #       render :action => "help", :layout => "help"
 
156   #     end
 
157   #   end
 
158   #
 
159   # This will render the help action with the "help" layout instead of the controller-wide "weblog_standard" layout.
