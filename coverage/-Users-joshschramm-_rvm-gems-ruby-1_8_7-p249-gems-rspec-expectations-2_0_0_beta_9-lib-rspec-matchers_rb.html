22   # == Predicates
 
23   #
 
24   # In addition to those Expression Matchers that are defined explicitly, RSpec will
 
25   # create custom Matchers on the fly for any arbitrary predicate, giving your specs
 
26   # a much more natural language feel. 
27   #
 
28   # A Ruby predicate is a method that ends with a "?" and returns true or false.
 
29   # Common examples are +empty?+, +nil?+, and +instance_of?+.
 
30   #
 
31   # All you need to do is write +should be_+ followed by the predicate without
 
32   # the question mark, and RSpec will figure it out from there. For example:
 
33   #
 
34   #   [].should be_empty => [].empty? #passes
 
35   #   [].should_not be_empty => [].empty? #fails
 
36   #
 
37   # In addtion to prefixing the predicate matchers with "be_", you can also use "be_a_"
 
38   # and "be_an_", making your specs read much more naturally:
 
39   #
 
40   #   "a string".should be_an_instance_of(String) =>"a string".instance_of?(String) #passes
 
41   #
 
42   #   3.should be_a_kind_of(Fixnum) => 3.kind_of?(Numeric) #passes
 
43   #   3.should be_a_kind_of(Numeric) => 3.kind_of?(Numeric) #passes
 
44   #   3.should be_an_instance_of(Fixnum) => 3.instance_of?(Fixnum) #passes
 
45   #   3.should_not be_instance_of(Numeric) => 3.instance_of?(Numeric) #fails
 
46   #
 
47   # RSpec will also create custom matchers for predicates like +has_key?+. To
 
48   # use this feature, just state that the object should have_key(:key) and RSpec will
 
49   # call has_key?(:key) on the target. For example:
 
50   #
 
51   #   {:a => "A"}.should have_key(:a) => {:a => "A"}.has_key?(:a) #passes

57   # == Custom Matchers
 
58   #
 
59   # When you find that none of the stock Expectation Matchers provide a natural
 
60   # feeling expectation, you can very easily write your own using RSpec's matcher
 
61   # DSL or writing one from scratch.
 
62   #
 
63   # === Matcher DSL
 
64   #
 
65   # Imagine that you are writing a game in which players can be in various
 
66   # zones on a virtual board. To specify that bob should be in zone 4, you
 
67   # could say:
 
68   #
 
69   #   bob.current_zone.should eql(Zone.new("4"))
 
70   #
 
71   # But you might find it more expressive to say:
 
72   #
 
73   #   bob.should be_in_zone("4")
 
74   #
 
75   # and/or
 
76   #
 
77   #   bob.should_not be_in_zone("3")
 
78   #
 
79   # You can create such a matcher like so:
 
80   #
 
81   #   RSpec::Matchers.define :be_in_zone do |zone|
 
82   #     match do |player|
 
83   #       player.in_zone?(zone)
 
84   #     end
 
85   #   end
 
86   #
 
87   # This will generate a <tt>be_in_zone</tt> method that returns a matcher
 
88   # with logical default messages for failures. You can override the failure
 
89   # messages and the generated description as follows:
 
90   #
 
91   #   RSpec::Matchers.define :be_in_zone do |zone|
 
92   #     match do |player|
 
93   #       player.in_zone?(zone)
 
94   #     end
 
95   #     failure_message_for_should do |player|
 
96   #       # generate and return the appropriate string.
 
97   #     end
 
98   #     failure_message_for_should_not do |player|
 
99   #       # generate and return the appropriate string.
 
100   #     end
 
101   #     description do
 
102   #       # generate and return the appropriate string.
 
103   #     end
 
104   #   end
 
105   #
 
106   # Each of the message-generation methods has access to the block arguments
 
107   # passed to the <tt>create</tt> method (in this case, <tt>zone</tt>). The
 
108   # failure message methods (<tt>failure_message_for_should</tt> and
 
109   # <tt>failure_message_for_should_not</tt>) are passed the actual value (the
 
110   # receiver of <tt>should</tt> or <tt>should_not</tt>).
